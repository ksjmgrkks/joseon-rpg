# 조선시대 도트 횡스크롤 RPG — AI 기반 전체 제작 로드맵

> AI로 에셋과 작업 대부분을 처리하되, 사람이 방향성·품질·통일성을 잡는 **"센타우로스(Human-AI 하이브리드)"** 방식을 전제로 한 로드맵입니다. 순수 AI 자동화로는 도트 일관성과 조선시대 고유 비주얼이 무너지므로, 단계마다 "AI가 하는 일 / 사람이 잡는 일"을 구분했습니다.

---

## 0. 핵심 원칙 (먼저 머리에 박아둘 것)

1. **레퍼런스 우선(Reference-first).** 텍스트 프롬프트만으로는 조선풍이 중국풍·사무라이풍으로 샙니다. 캐릭터 베이스 1~2장을 먼저 완성하고, 그걸 레퍼런스로 전체를 확장합니다.
2. **스타일을 먼저 잠그고 양산은 나중에.** 첫 캐릭터의 해상도·팔레트·아웃라인 규칙·픽셀 밀도를 확정하기 전에는 대량 생성하지 않습니다.
3. **AI 출력 = 초안.** 스프라이트는 Aseprite에서 손보정하는 단계를 파이프라인에 상수로 넣습니다. 특히 애니메이션 프레임 간 어긋남, 서브픽셀 깨짐 보정.
4. **수직 슬라이스부터.** 한 캐릭터가 한 맵에서 걷고·싸우고·대화하는 "10분짜리 완성 구간"을 먼저 만들고, 그게 재밌으면 양산합니다.

## 권장 도구 스택

| 영역 | 도구 | 역할 |
|---|---|---|
| 엔진 | **Godot 4.x** | 무료·경량·픽셀퍼펙트 2D에 최적 |
| 스프라이트/타일 생성 | **PixelLab** | 스켈레톤 기반 애니메이션, 방향 뷰, 타일셋, Aseprite 플러그인 |
| 스타일 일관성 대안 | **a1.art** | 안정적 스타일·디테일한 2D 에셋 |
| 도트 보정 | **Aseprite** | AI 출력 손보정, 팔레트/프레임 정리 |
| 코드 | **Claude Code / Copilot** | GDScript 로직, 구조 코드, 시스템 |
| 사운드 | AI 음악/효과음 생성기 | BGM, 효과음 초안 |
| 기획·리서치 | **Perplexity / Claude** | 사료 조사, 밸런싱, 레퍼런스 정리 |

## Phase 0 — 사전 설정 & 스타일 바이블 (1~2주)

**목표: 게임의 "얼굴"을 확정한다.**

- [ ] 게임 컨셉 1장 요약: 장르(액션 RPG/턴제), 시대 배경(조선 초/중/말), 톤(진지/코믹/괴담)
- [ ] **해상도 규칙 확정**: 캐릭터 높이(예: 32px / 48px), 타일 크기(16x16 or 32x32), 픽셀당 화면 비율
- [ ] **팔레트 확정**: 단청·먹·한지 톤을 살린 제한 팔레트(예: 24~32색). 팔레트를 고정해야 AI 출력이 통일됩니다.
- [ ] **주인공 베이스 스프라이트 완성** (가장 중요): AI로 여러 장 생성 → 가장 조선풍인 것 선택 → Aseprite에서 다듬어 "정답 레퍼런스" 1장 확정
- [ ] 스타일 바이블 문서화: 아웃라인 유무, 그림자 처리, 명암 단계 수 등

> **조선시대 함정 주의:** 한복·갓·도포·저고리·치마, 한옥·기와·단청, 양반/상민 복식 차이는 AI가 잘 모릅니다. Phase 0에서 실제 사진·사료 레퍼런스를 모아두고, 생성 시 image-to-image로 강제하세요.

## Phase 1 — 수직 슬라이스 / 플레이 가능한 프로토타입 (2~4주)

**목표: 한 구간이 "재밌는지" 검증한다. 아직 양산 금지.**

- [ ] Godot 프로젝트 세팅: 픽셀퍼펙트 카메라, 입력 맵, 씬 구조
- [ ] 주인공 기본 애니메이션 세트 생성(대기/걷기/달리기/점프/공격) — PixelLab 스켈레톤 기반
- [ ] Aseprite로 프레임 보정 → Godot로 임포트, AnimationPlayer 연결
- [ ] 횡스크롤 이동·점프·기본 전투 구현 (Claude Code로 GDScript 작성)
- [ ] 타일셋 1세트 생성(한옥 마을 or 산길) → 테스트 맵 1개 제작
- [ ] NPC 1명 + 대화창 시스템 (분기형으로 시작)
- [ ] **플레이 테스트**: 10분이 재밌는가? 아니면 컨셉/조작감부터 수정

---

## Phase 2 — 핵심 시스템 구축 (4~8주)

**목표: 게임의 뼈대 시스템을 완성한다. AI 코드 생성이 가장 빛나는 구간.**

- [ ] 전투 시스템 확장: 적 AI, 피격/넉백, 콤보 또는 스킬
- [ ] 인벤토리 / 아이템 / 장비 시스템
- [ ] 세이브/로드(직렬화) — AI가 보일러플레이트로 빠르게 처리
- [ ] 대화·퀘스트 시스템 (분기 + 플래그 관리)
- [ ] UI/HUD: 체력, 메뉴, 대화창 — 조선풍 한지/먹 테마 픽셀 UI
- [ ] 적 캐릭터 3~5종 생성(예: 산적, 도깨비, 호환 등 조선 괴담 소재 활용)

> **NPC 대화 분기점:** 미리 쓴 분기형 대사로 갈지, LLM 연동(맥락 인식·기억형 NPC)으로 갈지 여기서 결정. LLM을 쓴다면 사극 어투 가이드를 프롬프트에 강하게 넣어야 톤이 유지됩니다.

---

## Phase 3 — 콘텐츠 양산 (8~16주, 가장 길고 반복적)

**목표: Phase 0에서 잠근 스타일로 분량을 채운다.**

- [ ] 스테이지/맵 양산 (마을 → 산 → 관아 → 던전 등) — 타일셋 재활용 + 변주
- [ ] 적/보스 추가 생성 (레퍼런스 기반으로 스타일 통일)
- [ ] 메인 스토리 + 사이드 퀘스트 작성 (Claude로 초안, 사람이 톤 다듬기)
- [ ] BGM·효과음 양산 (국악 소재 + AI 생성, 지역별 테마)
- [ ] 컷신/이벤트 연출

> **이 구간의 진짜 작업은 "일관성 관리"입니다.** AI가 새 에셋을 뽑을 때마다 기존 팔레트·해상도·스타일에 맞는지 검수하고 보정하는 루프가 반복됩니다. 이 검수를 건너뛰면 게임이 "여기저기서 긁어온 느낌"이 됩니다.

---

## Phase 4 — 폴리시 & 출시 (4~8주)

**목표: 완성도와 마감.**

- [ ] 밸런싱(난이도·경제·전투) — 플레이 데이터 기반 조정
- [ ] 버그 수정, 최적화
- [ ] 타이틀/엔딩/메뉴 등 마무리 화면
- [ ] 빌드 & 플랫폼 결정 (PC/Steam, 모바일, 웹)
- [ ] 스토어 페이지·트레일러·스크린샷
- [ ] 클로즈드 테스트 → 피드백 반영 → 출시

---

## 조선시대 특화 체크리스트

- [ ] 복식: 양반(도포·갓) vs 상민(저고리·패랭이) vs 관리(관복) 구분이 스프라이트에서 드러나는가
- [ ] 건축: 한옥 기와·초가·담장·대문이 중국/일본풍과 섞이지 않았는가
- [ ] 문양/UI: 단청·태극·한지 질감 등 한국적 모티프 사용
- [ ] 적/몬스터: 도깨비, 구미호, 저승사자, 호환(호랑이) 등 한국 설화 소재
- [ ] 텍스트: 폰트가 픽셀 한글을 깨끗하게 렌더하는가 (한글 픽셀 폰트 별도 준비)
- [ ] 어투: 대사가 사극체로 통일됐는가

---

## 현실적인 주의사항

- **도트 일관성**은 AI의 알려진 약점입니다. 한 장에서 멀쩡하던 캐릭터가 애니메이션으로 가면 어긋나고, 픽셀 아트는 서브픽셀 수준에서 자주 깨집니다. → 손보정 단계를 일정에 반드시 포함.
- **저작권/라이선스**: 사용하는 AI 도구마다 상업적 이용 약관이 다릅니다. 출시 전 각 도구의 라이선스를 반드시 확인하세요.
- **AI 사용 정서**: 최근 업계 설문에서 생성형 AI에 대한 개발자 반발이 큽니다(약 절반이 부정적). 출시 시 AI 활용 범위를 어떻게 커뮤니케이션할지도 고려해두면 좋습니다.
